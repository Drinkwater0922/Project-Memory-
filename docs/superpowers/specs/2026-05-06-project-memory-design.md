# Project Memory for Mac - 产品机会与设计规格

日期：2026-05-06
状态：待评审草稿

## 1. 核心判断

Project Memory 是一个本地优先的 Mac 应用，帮助开发者、产品型创始人和研究型创作者，从分散的文件、笔记、浏览网页、PDF 和 Git 历史中恢复项目上下文。

它不应该被定位成另一个“第二大脑”，也不应该做成通用的本地 RAG 工具。更锋利的切口是项目状态恢复：

> “这个项目我上次做到哪了？为什么停在这里？哪些资料重要？下一步该做什么？”

第一版要验证的不是“能不能搜索所有文件”，而是用户是否愿意每天打开它，因为它真的减少了跨项目切换时的认知加载成本。

## 2. 目标用户

早期核心用户：

- 同时推进多个产品想法的独立开发者和 solo founder。
- 把 PRD、研究链接、代码和笔记分散在多个地方的产品型工程师。
- 使用 Markdown、PDF、网页保存和 Git 管理写作或研究资料的创作者。

第一版只服务一个人在一台 Mac 上的个人工作流。团队共享、企业权限和管理员级 SaaS 连接器都放到后续阶段。

## 3. 产品定位

一句话定位：

> 一个本地优先的 Mac 应用，记住你的项目上次停在哪里。

产品不是：

- 不是笔记应用。
- 不是文档管理器。
- 不是“所有文件问答”的通用聊天机器人。

产品是：

- 一个本地项目记忆层，把项目活动和资料沉淀成时间线、每日简报和带来源证据的回答。

核心差异化：

- 以项目为中心，而不是以文件为中心。
- 主动生成简报，而不是只等用户搜索。
- 默认本地优先。
- 回答必须带来源证据，而不是自由发挥式总结。
- 同时覆盖代码、产品资料和写作/研究资料。

## 4. MVP 范围

MVP 要跑通一个窄但完整的闭环：

1. 用户选择本地项目文件夹。
2. 应用索引支持的文件和活动。
3. 应用识别项目，或让用户手动创建项目。
4. 应用构建项目时间线，包括文件变化、资料、笔记和 Git 活动。
5. 应用生成每日项目简报。
6. 用户可以围绕某个项目提问，并获得带来源的回答。
7. 用户可以纠正项目归属和关键事实。

### MVP 数据源

必须支持：

- 用户显式选择的本地文件夹。
- Markdown 文件，包括 Obsidian vault。
- PDF。
- 网页保存或浏览器扩展捕获内容。
- Git repo，包括 commit、branch、diff 和最近修改文件。

如果成本较低，可以支持：

- 纯文本文件。
- HTML 文件。
- 通过本地解析器转换后的 Word 文档。

明确不做：

- 全量聊天记录接入。
- Apple Health 或个人健康数据。
- 邮箱收件箱连接器。
- 默认读取私有应用缓存。
- 飞书、钉钉、企业微信的深度 API 同步。
- 多人协作。
- 自动执行任务。

## 5. 核心体验

### 主窗口

主窗口第一屏应该是项目记忆，而不是空白聊天框。

主要视图：

- 今日：跨活跃项目的每日简报。
- 项目：项目卡片和项目时间线。
- 来源：已索引的文件、PDF、网页、笔记和 repo。
- 提问：围绕项目的问答入口。
- 设置：隐私、模型、数据源和同步控制。

### 项目卡片

每个项目卡片展示：

- 当前状态：用一段话说明项目目前推进到哪里。
- 最近进展：commit、文档变更、新笔记、保存网页、PDF 新增等。
- 新增资料：最近加入或新近变得相关的资料。
- 未闭环问题：TODO、未解决笔记、不清楚的决策、重复研究的问题。
- 决策记录：重要决策及其来源链接。
- 下一步建议：1-3 个具体行动，每个行动都要有依据。

### 每日简报

每日简报回答：

- 最近哪些项目有推进？
- 昨天或上次简报后发生了什么变化？
- 哪些项目看起来停滞了？
- 哪些资料新增了但还没有归到项目？
- 哪些 TODO 或开放问题可能被遗忘了？
- 今天最值得继续的 1-3 件事是什么？

### 项目问答

第一版支持的问题类型：

- “这个项目我上次做到哪了？”
- “上周之后发生了什么变化？”
- “我当时为什么做这个决策？”
- “哪些资料支持这个 PRD 或文章段落？”
- “我之前是不是研究过这个问题？”
- “把当前项目上下文整理成实现计划或写作大纲。”

所有回答都必须附带来源引用，指向文件、网页、笔记、commit 或 PDF。

## 6. 隐私与信任模型

默认承诺：

- 用户选择的本地数据默认留在 Mac 上，除非用户主动启用云模型。
- 本地索引和元数据存储在设备上。
- 第一次使用外部模型前，应用必须清楚说明哪些内容会被发送出去。
- 云模型调用只发送相关片段，不上传整个磁盘或整个项目。
- 在可行的情况下，发送前对敏感字段做脱敏。
- MVP 验证工作流后，再支持 Ollama 或其他本地模型端点。

信任原则：

- 不静默扫描整个磁盘。
- 不默认读取私有应用缓存。
- 优先使用明确的文件夹授权和明确的浏览器捕获。
- 用户可以查看、移除和重新归类来源。
- 所有关键结论都展示来源链接。

## 7. 国内办公工具策略

飞书、钉钉、企业微信对目标用户很重要，但不应该作为 MVP 的基础依赖。

产品采用三阶段策略：

### 阶段 1：本地文件与用户主动捕获

- 监听用户选择的本地文件夹，包括下载目录和同步目录。
- 识别从飞书、钉钉、企业微信、腾讯文档等工具导出或下载的文件。
- 提供浏览器扩展或分享动作，把当前网页文档保存到 Project Memory。

### 阶段 2：飞书连接器

- 优先做飞书连接器，因为飞书文档和知识库更贴近项目记忆场景。
- 支持“把这个文档/文件夹/知识库节点加入项目记忆”的显式操作。
- 在没有管理员权限前，不承诺全量企业同步。

### 阶段 3：钉钉与企业微信连接器

- 把钉钉和企业微信作为团队版或企业版能力。
- 预期会遇到管理员授权、接口差异和文件权限复杂度。
- 连接器边界必须显式、可审计。

## 8. 系统架构

MVP 推荐架构：

- Mac 原生外壳：SwiftUI menu bar 应用 + 主窗口。
- 本地服务：负责后台索引、解析、embedding 任务和简报生成。
- 存储：SQLite 存储元数据、事件时间线、项目归属和来源记录。
- 文件缓存：本地保存抽取文本、网页快照和缩略图。
- 搜索：关键词检索 + 向量检索的混合搜索。
- 模型层：抽象出模型 provider，支持本地模型端点和云 LLM provider。

### 主要组件

Source Manager：

- 跟踪用户选择的文件夹、repo 和浏览器捕获内容。
- 发出文件变化和 repo 变化事件。

Parser Pipeline：

- 从 Markdown、PDF、HTML/网页捕获和 Git 历史中抽取文本与元数据。
- 统一时间戳、标题、作者、路径、URL 和项目线索。

Project Resolver：

- 把来源和事件归属到项目。
- 结合路径规则、Git repo 边界、用户显式纠正和语义相似度。

Memory Store：

- 存储来源、文本块、embedding、时间线事件、项目事实、决策和用户纠正。

Brief Generator：

- 生成每日简报和项目级摘要。
- 基于时间线变化和检索到的证据生成内容。

Answer Engine：

- 处理项目范围内的问题。
- 先检索证据，再生成带引用的回答。

Review UI：

- 让用户确认或纠正项目归属、决策和下一步建议。

## 9. 数据流

1. 用户选择文件夹、Obsidian vault、网页保存目录或 Git repo。
2. Source Manager 注册数据源并开始增量监听。
3. Parser Pipeline 抽取文本、元数据和事件。
4. Project Resolver 把资料归属到已有项目或新项目。
5. Memory Store 持久化来源记录、文本块、时间线事件和 embedding。
6. Brief Generator 根据最近变化生成项目简报和每日简报。
7. UI 展示带来源证据的简报。
8. 用户纠正会更新归属规则和项目记忆。
9. Answer Engine 使用项目范围内的检索结果回答问题。

## 10. 错误处理与边界情况

权限不足：

- 展示数据源级别错误，并说明如何重新授权文件夹访问。
- 继续索引其他数据源。

不支持的文件：

- 只保存元数据。
- 标记抽取状态为不支持。

大型 PDF 或损坏文件：

- 尽可能抽取部分文本。
- 展示抽取警告，但不阻塞整个索引。

超大 Git 历史：

- 先处理最近 commit 和 working tree 状态。
- 历史记录逐步回填。

项目归属错误：

- 提供一键重新归属。
- 把纠正保存成规则或学习信号。

云模型关闭：

- 保留搜索、时间线和尽可能基础的确定性摘要。
- 清楚标记需要模型的功能不可用或仅支持本地模型。

## 11. 成功指标

MVP 验证应该关注用户行为，而不是功能数量。

激活：

- 用户至少连接一个文件夹和一个 Git repo。
- 应用至少创建一条有用的项目时间线。
- 用户打开第一份生成的简报。

留存：

- 用户第一周至少 3 次打开应用或 menu bar 简报。
- 用户至少 2 次提问“我上次做到哪了”或类似问题。

质量：

- 用户认为每日简报有用的比例至少达到 60%。
- 关键结论的来源引用正确。
- 项目归属纠正率随时间下降。

商业信号：

- 用户明确表示愿意为跨项目恢复上下文付费。
- 最强价值时刻发生在开始工作前，而不是索引文件后。

## 12. 主要风险

掉进通用 RAG 陷阱：

- 如果第一屏变成“问你的文件”，产品会失去差异化。
- 必须保持“今日”和“项目时间线”为主要界面。

项目归属质量差：

- 如果资料经常归错项目，用户会快速失去信任。
- 第一版先使用明确的文件夹/repo 到项目映射，再逐步增加自动化。

解析复杂度高：

- 本地文档格式很脏。
- 先深度支持少数格式，再增加更多浅层连接器。

隐私表述模糊：

- 本地优先产品必须非常明确地说明数据流向。
- 模型 provider 设置和用户同意必须可见、可理解。

Mac app 开发复杂：

- 原生开发、文件权限、后台服务和浏览器扩展都会增加成本。
- 先做薄但可信的桌面应用，再追求复杂 UI。

## 13. 推荐构建顺序

阶段 0：Concierge 原型

- 手动让应用或脚本指向一个用户的文件夹、Markdown vault 和 Git repo。
- 生成静态每日项目简报。
- 验证简报是否有用到值得每天阅读。

阶段 1：本地索引与项目时间线

- 文件夹选择。
- Markdown/PDF/网页捕获解析。
- Git 最近活动解析。
- SQLite 元数据存储。
- 手动创建项目和来源归属。

阶段 2：每日简报与问答

- 项目时间线 UI。
- 每日简报生成。
- 带引用的项目范围问答。
- 用户纠正闭环。

阶段 3：浏览器捕获与 Obsidian 优化

- 浏览器扩展或分享工作流。
- 更好的 Markdown/Obsidian 双链处理。
- 网页快照保存。

阶段 4：隐私与模型选项

- 云模型 provider 设置。
- 本地模型端点选项。
- 发送外部模型前的脱敏和预览。

阶段 5：办公工具连接器

- 飞书显式文档/文件夹连接器。
- 钉钉和企业微信等企业权限策略清晰后再做。

## 14. 已确定的第一版决策

第一版 dogfood 的关键决策如下：

- 技术形态：SwiftUI 原生 Mac 应用。
- 核心能力：每日简报和项目问答第一轮一起做。
- 模型 provider：OpenRouter API key，由用户在本地设置中配置。
- 浏览器捕获：进入第一轮 dogfood。
- 初始验证用户：创始人自己先 dogfood。

## 15. 建议

第一版 dogfood 围绕一个核心承诺构建：

> “在我重新开始一个项目之前，Project Memory 告诉我上次停在哪里，并展示依据。”

应用首先要在一个 Git repo、一个笔记文件夹和少量网页保存资料上变得有用。如果在这个窄上下文里都无法生成真正有帮助的简报，那么增加飞书、钉钉、企业微信、健康数据或更多文件类型也解决不了产品问题。
